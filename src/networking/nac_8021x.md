# 802.1X / RADIUS / Network Access Control

## Overview

**802.1X** is port-based network access control: a device must *authenticate before it gets
any network access at all*. Plug into a switch port or associate to Wi-Fi, and the network
holds you in a "closed" state — only authentication traffic passes — until you prove who you
are. Then the port opens (and often you're assigned a VLAN/policy). This is how enterprise
wired and Wi-Fi (WPA2/3-Enterprise) keep unauthorized devices off the LAN.

```
The cast (the "AAA triangle"):
  Supplicant      the client device (laptop, phone) proving its identity
  Authenticator   the switch or Wi-Fi AP — the gatekeeper enforcing the port state
  Authentication  the RADIUS server (e.g. FreeRADIUS, Cisco ISE) that makes the decision
     Server       against a directory (LDAP/AD) or cert CA

  802.1X = the framing (EAP over LAN) between supplicant↔authenticator
  RADIUS = how the authenticator asks the auth server
  EAP    = the actual authentication method carried inside
```

## Why 802.1X

```
  ✓ Authenticates the DEVICE/USER at the edge, before an IP is even assigned (DHCP)
  ✓ Dynamic VLAN / ACL assignment per identity (see ethernet_vlan.md)
  ✓ Centralized policy + logging via RADIUS (no per-switch config of users)
  ✓ Defeats "just plug a rogue device into the wall jack" attacks
  ✗ Needs supplicant support + PKI/credentials management
  ✗ Headless devices (printers, cameras) need MAB fallback (weak — MAC is spoofable)
```

## How it works

### Port states & the EAP exchange

```
  Supplicant            Authenticator (switch)            RADIUS server
   (laptop)               port = UNAUTHORIZED
      |  EAPOL-Start  ───►  |  (only EAPOL allowed)
      |  ◄── EAP-Request/Identity                            |
      |  EAP-Response/Identity ──►  | ─ RADIUS Access-Request ─► |
      |                             |                            | (EAP method runs
      |  ◄══════ EAP method messages tunneled both ways ═══════► |   end-to-end:
      |                             |                            |   supplicant ↔ RADIUS;
      |                             | ◄─ RADIUS Access-Accept ── |   switch just relays)
      |  ◄── EAP-Success            |                            |
      |                       port = AUTHORIZED                  |
      |                       (+ VLAN/ACL from RADIUS attrs)     |
      | ─ DHCP, normal traffic ──►  |
```

Key point: the EAP conversation is **end-to-end between the supplicant and the RADIUS
server**. The switch is a blind relay — it never sees passwords or private keys, it just
enforces the final Accept/Reject and applies returned attributes.

### EAPOL vs RADIUS encapsulation

```
  supplicant  --EAP over LAN (EAPOL)-->  switch  --EAP inside RADIUS-->  RADIUS server
              802.1X frames, L2                   UDP/1812 (auth), 1813 (acct)
```

## EAP methods (what actually authenticates)

```
EAP-TLS        mutual certificate auth. Strongest. Client + server both present certs
               (see tls_ssl.md). No passwords on the wire. Needs client-cert PKI.
PEAP           server cert builds a TLS tunnel, then password auth (MSCHAPv2) inside it.
               Common in AD shops; only the server needs a cert.
EAP-TTLS       like PEAP; flexible inner methods (PAP/CHAP/etc.).
EAP-FAST       Cisco; PAC-based tunnel instead of a cert.
EAP-MD5        legacy, no mutual auth, no key derivation — DON'T use for Wi-Fi.
MAB            MAC Authentication Bypass: not EAP. Fallback for devices with no supplicant;
               the switch sends the device MAC as the credential. Spoofable → low trust VLAN.
```

For Wi-Fi, the EAP exchange also derives keys used by WPA2/WPA3-Enterprise to encrypt the
air link — so EAP-TLS/PEAP both authenticate *and* bootstrap link encryption.

## RADIUS

RADIUS (Remote Authentication Dial-In User Service) is the AAA protocol the authenticator
speaks to the policy server.

```
AAA:
  Authentication  who are you?      (Access-Request → Access-Accept/Reject/Challenge)
  Authorization   what may you do?  (attributes in the Accept: VLAN, ACL, session timeout)
  Accounting      what did you do?  (Accounting-Request: start/stop, bytes, session id)

Transport: UDP 1812 (auth) / 1813 (accounting). Legacy 1645/1646.
Security:  shared secret hashes the Authenticator + hides the User-Password attribute only —
           the rest is cleartext. RadSec (RADIUS over TLS, TCP/2083) wraps it properly.
```

### Dynamic VLAN / policy assignment

The RADIUS Access-Accept can carry attributes that put the user on the right
[VLAN](ethernet_vlan.md) or apply an ACL automatically:

```
Tunnel-Type            = VLAN (13)
Tunnel-Medium-Type     = IEEE-802 (6)
Tunnel-Private-Group-ID = 20            ← put this port/user on VLAN 20
Filter-Id / downloadable ACL            ← apply an access policy
Session-Timeout, Termination-Action     ← force periodic reauth
```

This is how a contractor lands on a guest VLAN and an employee on the corporate VLAN from
the *same* wall jack or SSID — the identity decides the network.

## NAC: the bigger picture

802.1X is the enforcement mechanism; **NAC (Network Access Control)** is the broader policy
system around it — often adding **posture assessment** (is the OS patched? AV running?
disk encrypted?), profiling of headless devices, and guest onboarding. Solutions: Cisco
ISE, Aruba ClearPass, packetfence (open source). A common pattern:

```
  Authenticate (802.1X/EAP) → Assess posture → Authorize (VLAN/ACL) → Monitor (accounting)
  Failing devices get quarantined on a remediation VLAN instead of blocked outright.
```

## Verifying / tooling

```bash
# Linux supplicant (wpa_supplicant) for wired EAP-TLS — wired example
# /etc/wpa_supplicant/wired.conf
network={
    key_mgmt=IEEE8021X
    eap=TLS
    identity="host/laptop.corp.example"
    ca_cert="/etc/ssl/corp-ca.pem"
    client_cert="/etc/ssl/laptop.pem"
    private_key="/etc/ssl/laptop.key"
}
# run:
wpa_supplicant -i eth0 -D wired -c /etc/wpa_supplicant/wired.conf -d

# Test a RADIUS server directly (freeradius tools)
radtest user password radius-server:1812 0 sharedsecret
radclient radius-server:1812 auth sharedsecret <<< "User-Name=user,User-Password=pw"

# Switch-side (Cisco-style, for reference)
#   show authentication sessions interface gi1/0/5
#   show dot1x all
```

## Security notes & pitfalls

```
- Prefer EAP-TLS (cert-based, mutual) > PEAP/TTLS (password-in-tunnel) >> EAP-MD5 (never).
- ALWAYS validate the RADIUS server certificate on the supplicant, or PEAP/TTLS is MITM-able
  (rogue AP harvests MSCHAPv2 → offline crack). Pin the CA + server name.
- MAB is a fallback, not security — MACs are trivially spoofed; pin MAB devices to a
  low-trust VLAN with tight ACLs.
- Protect the RADIUS shared secret; use long random secrets and prefer RadSec (TLS).
- Plan a fail-open vs fail-closed policy for when RADIUS is unreachable.
- Combine with DHCP snooping / Dynamic ARP Inspection (see dhcp.md, arp.md) for L2 hardening.
```
