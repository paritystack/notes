# Embedded Rust (no_std)

## Overview

The rest of this book is a C and C++ story: hand-written [startup code](startup_code.md), `volatile` register pokes guarded by [MISRA rules](coding_standards.md), and bugs that surface as a [HardFault](hardfault_debugging.md) in the field. **Embedded Rust** asks a different question — what if the compiler could reject the data race, the use-after-free, and the double-configured pin *before* the binary is flashed? It targets the same Cortex-M / RISC-V / Xtensa parts as C, links with the same [linker scripts](linker_scripts.md), and is debugged over the same [SWD probe](jtag_swd.md), but moves a whole class of memory- and concurrency-safety bugs from runtime to compile time. This page covers the `no_std` model, the PAC → HAL → BSP crate stack, ownership applied to hardware, the two dominant concurrency frameworks (RTIC and Embassy), and where Rust does and doesn't pull its weight on a microcontroller. It assumes the general-purpose [Rust](../programming/rust.md) page for syntax and ownership basics.

```
   Where a memory/concurrency bug is caught
   ┌───────────────┬──────────────┬─────────────┬───────────┐
   │ Rust compiler │ Static       │ Test / HIL  │ FIELD     │
   │ (borrow check,│ analyzer     │ rig         │ (brick)   │
   │  typestate)   │ (clippy)     │             │           │
   └───────────────┴──────────────┴─────────────┴───────────┘
   C/C++ relies mostly on the right three columns; Rust shifts
   data races, use-after-free and aliasing into the first.
```

## The `no_std` Model

A microcontroller has no operating system, no heap by default, and no `println!`. Rust's standard library (`std`) assumes all three, so embedded crates opt out with `#![no_std]`, pulling in only `core` (and optionally `alloc` if you provide an allocator). You also opt out of the normal `main`/runtime startup with `#![no_main]`.

```rust
#![no_std]
#![no_main]

use cortex_m_rt::entry;
use panic_halt as _;          // links a #[panic_handler]; required in no_std

#[entry]
fn main() -> ! {              // never returns — there is nowhere to return to
    loop { }
}
```

What you give up versus hosted Rust, and the embedded replacement:

| `std` feature | `no_std` reality | Replacement |
|---------------|------------------|-------------|
| Heap (`Box`, `Vec`, `String`) | None by default | `heapless::Vec`, fixed capacity; or `alloc` + an allocator |
| `panic!` unwinding | No unwinder | A `#[panic_handler]` (`panic-halt`, `panic-probe`) |
| Startup / `main` runtime | None | `cortex-m-rt`'s `#[entry]`, reset vector, `.data`/`.bss` init |
| `println!` | No console | `defmt` over [RTT](rtt_semihosting.md), or `semihosting` |
| OS threads | None | [interrupts](interrupts.md), or an [RTOS](../rtos/README.md)-like framework |

`cortex-m-rt` is the Rust equivalent of the [C runtime / startup code](startup_code.md): it provides the reset handler, copies `.data` to RAM, zeroes `.bss`, lays out the [interrupt vector table](interrupts.md), and hands control to `#[entry]`.

## The Crate Stack: PAC → HAL → BSP

Where C has CMSIS device headers plus a vendor HAL, Rust has a three-layer tower of crates, each generated or built on the one below.

```
   ┌──────────────────────────────────────────────┐
   │ BSP   (board)   nucleo_f401re, microbit       │  pins named by silkscreen
   ├──────────────────────────────────────────────┤
   │ HAL   (chip)    stm32f4xx-hal, esp-hal        │  embedded-hal traits, typestate
   ├──────────────────────────────────────────────┤
   │ PAC   (svd2rust) stm32f4 / pac                 │  every register, type-safe
   ├──────────────────────────────────────────────┤
   │ cortex-m / riscv  core peripherals, NVIC, SCB  │
   └──────────────────────────────────────────────┘
```

- **PAC (Peripheral Access Crate)** — generated by `svd2rust` from the vendor's SVD file, the same XML that feeds [CMSIS](cmsis.md). Every register field becomes a typed method, so you write `gpioa.moder().modify(|_, w| w.moder5().output())` instead of `GPIOA->MODER |= (1 << 10)`. Illegal bit patterns won't compile.
- **HAL (Hardware Abstraction Layer)** — a per-chip crate implementing the portable **`embedded-hal`** traits (`OutputPin`, `SpiBus`, `I2c`, `DelayNs`). This is the layer that makes drivers reusable across vendors.
- **BSP (Board Support Package)** — optional top layer naming peripherals by the board's silkscreen (`board.leds.red`) instead of raw ports.

The payoff is **`embedded-hal`**: a sensor driver written against the `I2c` trait runs unchanged on STM32, nRF, ESP32, or RP2040. There is a large registry of these driver crates (`embedded-hal` ecosystem), which is the practical reason most teams reach for Rust on a new sensor-heavy board.

## Ownership Applied to Hardware

The borrow checker's rule — one mutable owner at a time — maps directly onto a truth C never enforces: a hardware peripheral is a singleton that two pieces of code must not configure simultaneously.

```rust
let dp = pac::Peripherals::take().unwrap();  // returns Some ONCE, None after
// a second take() returns None — the GPIO block can't be aliased
```

### Typestate: misconfiguration becomes a compile error

A pin's mode lives in its *type*, so calling a method invalid for the current mode simply doesn't compile — the [register-misconfiguration class of bug](coding_standards.md) is gone before runtime.

```rust
let gpioa = dp.GPIOA.split();
let mut led = gpioa.pa5.into_push_pull_output();   // type: Pin<Output>
led.set_high();                                    // OK

let button = gpioa.pa0.into_pull_up_input();       // type: Pin<Input>
// button.set_high();   // COMPILE ERROR: no set_high on an input pin
```

Contrast the C equivalent, where setting an output bit on a pin you forgot to configure as output is a silent no-op you debug with a scope.

## Concurrency: Interrupts, RTIC, and Embassy

`no_std` has no threads, so concurrency is built on [interrupts](interrupts.md). Sharing data between an ISR and `main` — the problem the [ring buffers](ring_buffers.md) page solves with `volatile` and memory barriers in C — is handled by the type system: shared state must live behind a `Mutex<RefCell<...>>` entered inside a **critical section**, so the compiler refuses to let you touch it with interrupts enabled.

Two frameworks dominate and you should know both:

- **RTIC** (Real-Time Interrupt-driven Concurrency) — a static, priority-based framework that maps tasks onto interrupt priorities and uses the NVIC as its scheduler. It does compile-time analysis to prove that shared-resource access is data-race-free *and* deadlock-free using the **Stack Resource Policy** (a [priority-ceiling protocol](../rtos/priority_inversion.md)) — so it sidesteps [priority inversion](../rtos/priority_inversion.md) by construction, no [RTOS](../rtos/README.md) kernel required. Zero-cost: tasks are just prioritized interrupt handlers.
- **Embassy** — an `async`/`await` executor for MCUs. Tasks are futures; awaiting a peripheral (a DMA transfer, a UART byte) yields the CPU so the core can sleep, making [tickless low-power](../rtos/tickless_idle.md) operation the default rather than an optimization. Excellent ergonomics for I/O-bound firmware (radios, sensors, [BLE](ble.md)).

```
   Pick a model:
   bare interrupts ── small, full manual control, you write the critical sections
   RTIC          ──── hard-real-time, static priorities, provably race-free
   Embassy       ──── async/await, many concurrent I/O tasks, low-power by default
```

## Memory & Cost

Rust's safety isn't free of footprint considerations, but it is free of a heap:

- **No allocator by default.** Use `heapless` for fixed-capacity `Vec`/`String`/`spsc::Queue` — the same "[no dynamic allocation after init](memory_management.md)" discipline MISRA *requests*, here enforced by not having `alloc` linked at all.
- **Monomorphization** (generics specialized per type) can bloat flash if you instantiate a big generic across many types — watch `.text` size, the analogue of C++ template bloat.
- **`unsafe` is still required** at the hardware boundary: raw DMA buffers, [cache coherency](cache_tcm.md), and writing your own PAC-less register access. Rust shrinks `unsafe` to a small, auditable surface rather than eliminating it — the value is that the other 95% of the firmware is checked.
- **Stack overflow** still corrupts memory unless you guard it; `flip-link` rearranges RAM so an overflow hits a guard region and faults instead of silently clobbering `.bss` — pair it with the [MPU](mpu.md) for hardware enforcement.

## Tooling

```bash
cargo install probe-rs-tools
cargo embed --release        # build, flash over SWD/JTAG, open an RTT console
```

- **`probe-rs`** — a pure-Rust replacement for OpenOCD/pyOCD that drives [SWD/JTAG](jtag_swd.md) probes (ST-Link, J-Link, CMSIS-DAP) for flashing and debugging, with a GDB server and VS Code integration. See [GDB for Embedded](gdb_embedded.md).
- **`defmt`** — a deferred-formatting logger: the device sends a compact ID + raw args over [RTT](rtt_semihosting.md) and the *host* does the string formatting, so logging costs a fraction of a `printf` and barely touches the firmware's flash/timing budget.
- **`cargo` + crates.io** — one build system and dependency manager, replacing the [Make/CMake/PlatformIO sprawl](build_systems.md). Cross-compilation is a target triple (`thumbv7em-none-eabihf`) plus a `.cargo/config.toml`.

## Pitfalls

1. **Reaching for `alloc` out of habit.** A heap on an MCU brings fragmentation and nondeterminism; prefer `heapless` fixed-capacity collections unless you have a clear reason.
2. **Fighting the borrow checker on shared state instead of using the right tool.** ISR-shared data wants `Mutex<RefCell<…>>` + critical section, or a framework (RTIC/Embassy) — not `static mut`, which is `unsafe` and now lints hard.
3. **Assuming `unsafe` disappeared.** DMA buffers, cache maintenance, and bare register access still need it; the win is keeping that surface small and reviewed, not zero.
4. **Ignoring monomorphization bloat.** Instantiating large generics across many types can blow the flash budget; measure `.text`.
5. **Treating `defmt` logs like `printf`.** Formatting happens host-side, so logging a value the host can't decode (mismatched `defmt` versions) yields garbage; keep host and target tooling in lockstep.
6. **No stack-overflow guard.** Rust doesn't bounds-check the stack; without `flip-link`/[MPU](mpu.md) an overflow is the same silent corruption as in C.
7. **Mixed maturity across vendors.** STM32, nRF, RP2040, and ESP32 HALs are strong; obscure parts may have thin or no crates — check the ecosystem before committing a board.
8. **Expecting C-identical codegen.** Bounds checks and panic paths add a little code/size; usually negligible, occasionally relevant in the tightest ISR — profile rather than assume.
