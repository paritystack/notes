My Notes

Design a Chat System (WhatsApp / Messenger / Slack)

Tests long-lived stateful connections, ordering, delivery semantics, group fan-out, and persistence.

1. Requirements

Functional

  • 1:1 chat: send text + media.
  • Group chat: up to N members (e.g., 256 for WhatsApp, 1K for Slack channels).
  • Delivery receipts: sent, delivered, read.
  • Online presence.
  • Message history with infinite scroll.
  • Offline → online: deliver missed messages.

Non-functional

  • 1B users, 500M DAU, 50M concurrent connections.
  • P99 same-region delivery < 500 ms.
  • Messages durable even if recipient offline.
  • 99.99% availability.

Out of scope (for this design)

  • E2E encryption (mention briefly).
  • Voice/video calls.

2. Capacity Estimation

DAU                  = 500M
Messages/user/day    = 40
Total msgs/day       = 20B → ~230K msg/s avg, ~1M peak
Concurrent conns     = 50M
Msg size             = ~200 B avg (text); media via blob store
Storage (text/year)  = 20B × 365 × 200 B = 1.5 PB/year

Bandwidth:
  Incoming msgs      = 1M × 200B = 200 MB/s
  Fan-out (avg 10 recipients/msg via groups) = 2 GB/s
  Heartbeats         = 50M × 100B / 30s ≈ 167 MB/s

3. API Design

Hybrid: WebSocket for real-time, REST for history/auth/admin.

# WebSocket frames (after handshake + auth)
→ { type:"send", client_msg_id, conv_id, body, ts }
← { type:"ack",  client_msg_id, server_msg_id, ts }
← { type:"deliver", conv_id, msg }
← { type:"read_receipt", conv_id, by, up_to_msg_id }

# REST
GET /v1/conversations?cursor=&limit=20
GET /v1/conversations/:id/messages?before_seq=&limit=50
POST /v1/conversations            # create 1:1 or group
POST /v1/conversations/:id/members
GET /v1/presence?user_ids=[...]

4. Data Model

users (PostgreSQL)
  user_id (Snowflake) PK, phone/email, name, ...

conversations (Cassandra)
  conv_id (Snowflake) PK
  type ('1:1' | 'group'), created_at, last_msg_id

conversation_members
  conv_id, user_id PK
  joined_at, role

messages (Cassandra; partition by conv_id, cluster by seq DESC)
  conv_id, seq (per-conv monotonic) PK
  msg_id (Snowflake)
  sender_id, body, media_url, sent_at, edited_at

user_inbox (Cassandra; partition by user_id, cluster by seq DESC)
  user_id, seq PK
  conv_id, msg_id, type ('msg'|'ack'|...)

read_state (Cassandra)
  conv_id, user_id PK
  last_read_seq

presence (Redis)
  key=user_id  val={status, last_seen, device_ids}
  TTL ~60s, refreshed by heartbeat

Per-conversation seq is the ordering anchor. Generated by a leader for that conv_id (e.g., the Cassandra coordinator + lightweight counter, or a sequencer per partition).

6. Deep Dives

A. Send Path: Reliable Delivery

1. Client → WS: { type:"send", client_msg_id, conv_id, body }
2. Edge → Msg svc (gRPC, includes user_id)
3. Msg svc:
     a. Idempotency: check client_msg_id (per sender, last 24h)
     b. Allocate server_msg_id (Snowflake), conv seq (atomic incr per conv)
     c. Persist message to Cassandra (durable)
     d. Persist into each recipient's user_inbox
     e. Publish to bus: topic = conv_id
4. Edge → Sender: { type:"ack", client_msg_id, server_msg_id }
5. Bus → all edges holding recipient conns → push { type:"deliver", msg }
6. Offline recipients: Push svc consumes bus → APNs/FCM

Why client_msg_id? Network drops between (1) and (4); client retries. Server dedups via client_msg_id index (24h TTL).

Why per-conv seq? Ordering. Two messages racing into the same conversation get monotonic seqs from a single sequencer. Group chats see consistent order.

B. Connection Tier Scaling

50M concurrent connections.

Per box (Linux + Go/Rust): 500K-1M conns
50M conns → 50-100 boxes minimum, 2× redundancy = 100-200 boxes

Sticky LB: hash user_id → edge pod. Cache locality + presence consistency.

Presence map: Redis hash user→{edge_id, last_seen}. TTL 60s, refreshed by heartbeats.

Edge → user routing: on incoming msg for user U, look up presence → publish to that edge’s topic. Edge process subscribes to its own topic.

Edge crash: presence entries expire; client reconnects elsewhere; pulls missed msgs from user_inbox since last delivered seq.

C. Group Chat Fan-out

1:1 fan-out is trivial (1 recipient). Groups need work.

Group sizeStrategy
≤ 256 (WhatsApp)Synchronous per-recipient inbox write, then publish
1K–100K (Slack channel, Discord)Don’t write to individual inboxes. Treat as a pub/sub topic — readers subscribe and pull on read
1M+ (broadcast channels)Like news feed: pull-on-read with a per-channel cache, treat like a “celebrity timeline”

Hybrid: small groups push to inboxes (fast read), large groups pull-on-read.

D. Ordering Guarantees

  • Within a conversation: total order (single-sequencer).
  • Across conversations for one user: best-effort. Two msgs in different convs arriving at the user are ordered by send time but not strictly causally.
  • Read receipts: stored separately; eventual consistency is fine.

E. Offline Delivery

Phone offline → message persisted to user_inbox, pushed via APNs/FCM.

Phone wakes → opens app → WS reconnects
Client sends: last_received_seq per conv
Server: SELECT * FROM user_inbox WHERE user_id=? AND seq > ? LIMIT N
Stream missed msgs over WS in order

User_inbox capped (e.g., 30 days). Beyond that, fetched from conv’s messages table.

F. Encryption (Brief)

E2E (Signal protocol / Double Ratchet):

  • Per-conv ratcheting keys per device.
  • Server stores ciphertext only.
  • Server never sees plaintext, but still sees metadata (sender, recipient, ts, size).
  • Adds: key distribution (X3DH), session setup, multi-device fanout.

E2E complicates server-side features (search, ML, anti-spam) — push to client.

9. Follow-up Questions

  • “How do you guarantee message ordering in a group?” — Single sequencer per conv: a Cassandra row counter or a per-conv leader. Each message gets a monotonic conv-seq before publish.
  • “What happens when a user is offline?” — Server still writes to their inbox and triggers a push notification (APNs/FCM). On next connect, client requests msgs since last_received_seq.
  • “How do read receipts work?” — Client sends read_up_to_seq per conv periodically. Sender’s edge receives via bus → updates UI. Stored in read_state table.
  • “How do you handle multiple devices per user?” — User has N device_ids. Inbox is per-user, deliveries fan out to all connected devices (via presence map → each device’s edge). Read-state syncs across devices.
  • “What if Kafka backs up?” — Producers detect lag (publish ack latency); apply backpressure to senders. Acks slow but messages still durable in DB.
  • “How does typing indicator scale?” — Don’t persist. Ephemeral pub/sub through bus only; if it drops, no harm. Throttle to 1 update/sec/user.
  • “What about a 1M-user group?” — Switch to pull-mode for that conv. Don’t write to inboxes; readers query conv messages directly, with caching.
  • “How would you build search?” — Per-user Elasticsearch index; either client-side (E2E) or server-side (non-E2E). Async indexing pipeline from Kafka.
  • “How do you delete a message?” — Tombstone in messages table, propagate via bus. Clients reconcile. For “delete for everyone”, retroactively rewrite inbox entries.
