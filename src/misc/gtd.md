# Getting Things Done (GTD)

## Overview

**Getting Things Done** (GTD) is a personal productivity method created by David Allen
(book of the same name, 2001). Its central premise is that the human mind is bad at
*storing* commitments but good at *processing* them: every open loop you keep in your head
quietly drains attention and creates a low-grade anxiety. GTD's fix is to move every
commitment out of your head into an external, trusted system, then run a defined workflow
that turns vague "stuff" into concrete next actions.

> "Your mind is for having ideas, not holding them." — David Allen

The payoff is a state Allen calls **"mind like water"**: because you trust the system to
surface the right thing at the right time, you can give whatever you're doing your full
attention without nagging worry about everything you're *not* doing.

```
The core idea:
  Open loops in your head   →  stress, half-remembered, re-decided constantly
  Open loops in a system    →  reviewed on schedule, decided once, acted on calmly
```

GTD is deliberately tool-agnostic — it works on index cards, a notebook, or any app. The
discipline is in the *workflow*, not the software.

## The Five Steps

GTD is a pipeline. Every input — an email, an idea in the shower, a meeting note, a bill —
flows through the same five stages. The key insight is to **separate collecting from
deciding from doing**; trying to do all three at once is what makes "stuff" pile up.

```
   CAPTURE          CLARIFY            ORGANIZE          REFLECT          ENGAGE
  ┌────────┐      ┌─────────┐        ┌──────────┐      ┌─────────┐     ┌─────────┐
  │ inbox  │ ───▶ │ what is  │ ────▶ │ put it    │ ──▶ │ weekly  │ ──▶ │ do the  │
  │ (catch │      │ it? is   │       │ where it  │     │ review  │     │ work    │
  │  all)  │      │ actionab-│       │ belongs   │     │ keeps   │     │ with    │
  │        │      │ le?      │       │ (a list)  │     │ trust   │     │ context │
  └────────┘      └─────────┘        └──────────┘      └─────────┘     └─────────┘
```

1. **Capture** — Collect everything that has your attention into trusted *inboxes* (a
   physical tray, a notes app, your email inbox). Capture must be frictionless, or you
   won't do it.
2. **Clarify** — Process each item: what is it, and is it actionable? (decision tree below).
3. **Organize** — Put the result where it belongs: a list, the calendar, reference, trash.
4. **Reflect** — Review the system regularly (daily glance, **weekly review**) so it stays
   current and you keep trusting it.
5. **Engage** — Use your lists to choose what to do *now*, based on context, time, energy,
   and priority.

## The Clarify Decision Tree

Clarifying is the heart of GTD — it converts ambiguous inbox items into decided outcomes.
Process strictly **top item first, one at a time, never back into the inbox.**

```
                        ┌──────────────┐
                        │  inbox item  │
                        └──────┬───────┘
                               ▼
                      Is it ACTIONABLE?
                  ┌────────────┴────────────┐
                 NO                         YES
                  │                          │
        ┌─────────┼──────────┐               ▼
        ▼         ▼          ▼      What's the NEXT ACTION?
     Trash    Someday/    Reference          │
              Maybe       (file it)  ┌────────┴─────────┐
                                     ▼                  ▼
                              Takes < 2 min?    Takes longer
                                     │                  │
                                    YES        ┌────────┴────────┐
                                     │         ▼                 ▼
                                  DO IT     Am I the         Defer it
                                  now      right person?         │
                                            │                    ▼
                                       ┌────┴────┐      ┌─────────────────┐
                                      NO        YES     │ single action?  │
                                       │         │      │  → Next Actions │
                                       ▼         │      │ multi-step?     │
                                   Delegate ─────┘      │  → Project +    │
                                  (Waiting For)         │    its next     │
                                                        │    action       │
                                                        └─────────────────┘
```

Two rules do most of the work:

- **The two-minute rule** — if the next action takes less than ~2 minutes, do it
  immediately; tracking it would cost more than doing it.
- **"What's the next action?"** — force every actionable item down to the next *physical,
  visible* step ("draft email to Sam", not "deal with the Sam thing"). Vague items are the
  number-one reason lists stall.

A **project**, in GTD terms, is *any outcome that needs more than one action* — not just
big initiatives. "Replace the kitchen tap" is a project. Every project must always have a
defined next action living on a Next Actions list.

## Organize: The Lists

The output of clarifying lands in a small set of well-defined buckets. Keep them separate —
mixing "things to do" with "things to remember" is what makes lists untrustworthy.

```
┌────────────────┬──────────────────────────────────────────────────────────┐
│ List           │ What lives here                                           │
├────────────────┼──────────────────────────────────────────────────────────┤
│ Next Actions   │ Concrete single steps, grouped by CONTEXT (see below)     │
│ Projects       │ Every outcome needing >1 action (just the names)          │
│ Waiting For     │ Delegated items + what you're waiting on, with a date    │
│ Calendar       │ ONLY time/day-specific commitments — no "to-do" clutter   │
│ Someday/Maybe  │ Not now, maybe later: ideas, trips, books, "one day"      │
│ Reference      │ Non-actionable info you may need: docs, notes, manuals    │
└────────────────┴──────────────────────────────────────────────────────────┘
```

**Contexts** are tags for *where/with what* an action can be done, so that when you have a
spare moment you can filter to what's actually possible right now:

```
@calls       things you can do with a phone
@computer    needs a screen / online
@errands     out-and-about, in town
@home        only doable at home
@office      only at work
@agenda:Sam  raise next time you talk to Sam
@waiting     blocked on someone else
```

Allen is strict about the **calendar**: it is "hard landscape" — only things that *must*
happen on a specific day or time go there (appointments, deadlines, day-specific reminders).
Everything else goes on Next Actions. This keeps the calendar trustworthy.

## The Weekly Review

The Weekly Review is the linchpin of GTD — the habit that keeps the whole system *trusted*
and therefore worth using. Skipping it is the most common way GTD quietly dies. Block ~30–90
minutes, same time each week, and run a checklist:

```
GET CLEAR
  [ ] Empty every inbox: physical tray, email, notes, voicemail, head
  [ ] Process all loose papers and captured notes to zero

GET CURRENT
  [ ] Review Next Actions — tick off done, delete dead ones
  [ ] Review Projects — does each still have a defined next action?
  [ ] Review Waiting For — chase anything stale
  [ ] Review Calendar — past week (loose ends?) and upcoming weeks

GET CREATIVE
  [ ] Review Someday/Maybe — promote anything now worth doing
  [ ] Add new ideas, projects, and bigger-picture goals
```

## Horizons of Focus

Day-to-day next actions are the ground floor. GTD also offers an "altitude" model so your
actions stay aligned with what actually matters — review the lower levels often, the higher
levels rarely.

```
  50,000 ft  PURPOSE & PRINCIPLES   why you exist; core values
  40,000 ft  VISION                 3–5 year picture of success
  30,000 ft  GOALS                  what to achieve in 1–2 years
  20,000 ft  AREAS OF RESPONSIBILITY  ongoing roles to maintain (health, team, finances)
  10,000 ft  PROJECTS               outcomes needing >1 action, ~weeks–months
   Ground    NEXT ACTIONS           the concrete steps you do right now
```

Most stress comes from a mismatch between levels — e.g. busy at the Ground floor while a
30,000 ft goal goes unserved. The Weekly Review handles the lower floors; a less frequent
review (monthly/quarterly/annually) handles the higher ones.

## Tools & Implementation

GTD is intentionally independent of any tool. Pick whatever you'll actually keep current:

```
Analog        notebook + index cards, paper inbox tray, manila folders for reference
Apps          Todoist, Things, TickTick, Microsoft To Do, OmniFocus (built around GTD)
Text/dev      Emacs org-mode, Obsidian/Notion with tags, plain Markdown + a folder per list
Email         turn the inbox into a true capture point; process it to empty, don't live in it
```

Whatever the tool, the non-negotiables are: a **frictionless capture** path, **lists that
map to the buckets above**, and a recurring **review** so the system stays trusted. The tool
matters far less than the discipline.

## Where this connects

- **PARA** (Projects / Areas / Resources / Archives) — Tiago Forte's organizing scheme for
  digital files; pairs naturally with GTD's Projects and Areas of Responsibility.
- **[Zettelkasten](zettelkasten.md) / note-taking** — GTD handles *actions*; a Zettelkasten handles *knowledge*.
  Keep reference and ideas out of your action lists and in a notes system.
- **Eisenhower matrix** (urgent/important) — a prioritization lens for *choosing* among Next
  Actions; GTD is mostly silent on prioritization beyond context/time/energy.
- [Pomodoro Technique](pomodoro.md) — a focus/timeboxing method for the *Engage* step, once
  you've decided what to work on.
- **Inbox Zero** — the email-specific application of Capture + Clarify (process to empty,
  don't use the inbox as a to-do list).
- Within this book, GTD habits sit well alongside a long-lived knowledge base; this very
  repo's index-and-cross-link approach mirrors GTD's "reference" bucket.

## Pitfalls

- **Capturing but never clarifying** — a giant inbox you never process is just a new pile of
  stress. Capture is worthless without regular clarifying.
- **Skipping the Weekly Review** — the system goes stale, you stop trusting it, and you
  revert to keeping things in your head. This is the #1 failure mode.
- **Fuzzy next actions** — "plan vacation" or "Mom" aren't actions. If a list item doesn't
  name a concrete physical step, you'll skip past it every time.
- **Projects with no next action** — a project that isn't represented by a live next action
  silently stalls; the Weekly Review exists partly to catch these.
- **Over-engineering the system** — endless tags, custom apps, and tweaking the setup is
  *procrastination disguised as productivity*. Favor the simplest system you'll maintain.
- **Cluttering the calendar** — putting "maybe today" to-dos on the calendar destroys trust
  in it; reserve it for genuinely day/time-specific commitments.
- **No discipline around contexts/time/energy** — having lists but always grabbing whatever's
  loudest defeats the point of organizing by context in the first place.
