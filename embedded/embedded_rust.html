Embedded Rust (no_std)

Overview

The rest of this book is a C and C++ story: hand-written startup code, volatile register pokes guarded by MISRA rules, and bugs that surface as a HardFault in the field. Embedded Rust asks a different question — what if the compiler could reject the data race, the use-after-free, and the double-configured pin before the binary is flashed? It targets the same Cortex-M / RISC-V / Xtensa parts as C, links with the same linker scripts, and is debugged over the same SWD probe, but moves a whole class of memory- and concurrency-safety bugs from runtime to compile time. This page covers the no_std model, the PAC → HAL → BSP crate stack, ownership applied to hardware, the two dominant concurrency frameworks (RTIC and Embassy), and where Rust does and doesn’t pull its weight on a microcontroller. It assumes the general-purpose Rust page for syntax and ownership basics.

   └───────────────┴──────────────┴─────────────┴───────────┘
   C/C++ relies mostly on the right three columns; Rust shifts

The no_std Model

A microcontroller has no operating system, no heap by default, and no println!. Rust’s standard library (std) assumes all three, so embedded crates opt out with #![no_std], pulling in only core (and optionally alloc if you provide an allocator). You also opt out of the normal main/runtime startup with #![no_main].

#![no_std]
#![no_main]

use cortex_m_rt::entry;
use panic_halt as _;          // links a #[panic_handler]; required in no_std

#[entry]
fn main() -> ! {              // never returns — there is nowhere to return to
    loop { }
}

What you give up versus hosted Rust, and the embedded replacement:

std featureno_std realityReplacement
Heap (Box, Vec, String)None by defaultheapless::Vec, fixed capacity; or alloc + an allocator
panic! unwindingNo unwinderA #[panic_handler] (panic-halt, panic-probe)
Startup / main runtimeNonecortex-m-rt’s #[entry], reset vector, .data/.bss init
println!No consoledefmt over RTT, or semihosting
OS threadsNoneinterrupts, or an RTOS-like framework

cortex-m-rt is the Rust equivalent of the C runtime / startup code: it provides the reset handler, copies .data to RAM, zeroes .bss, lays out the interrupt vector table, and hands control to #[entry].

The Crate Stack: PAC → HAL → BSP

Where C has CMSIS device headers plus a vendor HAL, Rust has a three-layer tower of crates, each generated or built on the one below.

   ┌──────────────────────────────────────────────┐
   │ BSP   (board)   nucleo_f401re, microbit       │  pins named by silkscreen
   ├──────────────────────────────────────────────┤
   │ HAL   (chip)    stm32f4xx-hal, esp-hal        │  embedded-hal traits, typestate
   ├──────────────────────────────────────────────┤
   │ PAC   (svd2rust) stm32f4 / pac                 │  every register, type-safe
   ├──────────────────────────────────────────────┤
   │ cortex-m / riscv  core peripherals, NVIC, SCB  │
   └──────────────────────────────────────────────┘
  • PAC (Peripheral Access Crate) — generated by svd2rust from the vendor’s SVD file, the same XML that feeds CMSIS. Every register field becomes a typed method, so you write gpioa.moder().modify(|_, w| w.moder5().output()) instead of GPIOA->MODER |= (1 << 10). Illegal bit patterns won’t compile.
  • HAL (Hardware Abstraction Layer) — a per-chip crate implementing the portable embedded-hal traits (OutputPin, SpiBus, I2c, DelayNs). This is the layer that makes drivers reusable across vendors.
  • BSP (Board Support Package) — optional top layer naming peripherals by the board’s silkscreen (board.leds.red) instead of raw ports.

The payoff is embedded-hal: a sensor driver written against the I2c trait runs unchanged on STM32, nRF, ESP32, or RP2040. There is a large registry of these driver crates (embedded-hal ecosystem), which is the practical reason most teams reach for Rust on a new sensor-heavy board.

Ownership Applied to Hardware

The borrow checker’s rule — one mutable owner at a time — maps directly onto a truth C never enforces: a hardware peripheral is a singleton that two pieces of code must not configure simultaneously.

#![allow(unused)]
fn main() {
let dp = pac::Peripherals::take().unwrap();  // returns Some ONCE, None after
// a second take() returns None — the GPIO block can't be aliased
}

Typestate: misconfiguration becomes a compile error

A pin’s mode lives in its type, so calling a method invalid for the current mode simply doesn’t compile — the register-misconfiguration class of bug is gone before runtime.

#![allow(unused)]
fn main() {
let gpioa = dp.GPIOA.split();
let mut led = gpioa.pa5.into_push_pull_output();   // type: Pin<Output>
led.set_high();                                    // OK

let button = gpioa.pa0.into_pull_up_input();       // type: Pin<Input>
// button.set_high();   // COMPILE ERROR: no set_high on an input pin
}

Contrast the C equivalent, where setting an output bit on a pin you forgot to configure as output is a silent no-op you debug with a scope.

Concurrency: Interrupts, RTIC, and Embassy

no_std has no threads, so concurrency is built on interrupts. Sharing data between an ISR and main — the problem the ring buffers page solves with volatile and memory barriers in C — is handled by the type system: shared state must live behind a Mutex<RefCell<...>> entered inside a critical section, so the compiler refuses to let you touch it with interrupts enabled.

Two frameworks dominate and you should know both:

  • RTIC (Real-Time Interrupt-driven Concurrency) — a static, priority-based framework that maps tasks onto interrupt priorities and uses the NVIC as its scheduler. It does compile-time analysis to prove that shared-resource access is data-race-free and deadlock-free using the Stack Resource Policy (a priority-ceiling protocol) — so it sidesteps priority inversion by construction, no RTOS kernel required. Zero-cost: tasks are just prioritized interrupt handlers.
  • Embassy — an async/await executor for MCUs. Tasks are futures; awaiting a peripheral (a DMA transfer, a UART byte) yields the CPU so the core can sleep, making tickless low-power operation the default rather than an optimization. Excellent ergonomics for I/O-bound firmware (radios, sensors, BLE).
   Pick a model:
   bare interrupts ── small, full manual control, you write the critical sections
   RTIC          ──── hard-real-time, static priorities, provably race-free
   Embassy       ──── async/await, many concurrent I/O tasks, low-power by default

Memory & Cost

Rust’s safety isn’t free of footprint considerations, but it is free of a heap:

  • No allocator by default. Use heapless for fixed-capacity Vec/String/spsc::Queue — the same “no dynamic allocation after init” discipline MISRA requests, here enforced by not having alloc linked at all.
  • Monomorphization (generics specialized per type) can bloat flash if you instantiate a big generic across many types — watch .text size, the analogue of C++ template bloat.
  • unsafe is still required at the hardware boundary: raw DMA buffers, cache coherency, and writing your own PAC-less register access. Rust shrinks unsafe to a small, auditable surface rather than eliminating it — the value is that the other 95% of the firmware is checked.
  • Stack overflow still corrupts memory unless you guard it; flip-link rearranges RAM so an overflow hits a guard region and faults instead of silently clobbering .bss — pair it with the MPU for hardware enforcement.

Tooling

cargo install probe-rs-tools
cargo embed --release        # build, flash over SWD/JTAG, open an RTT console
  • probe-rs — a pure-Rust replacement for OpenOCD/pyOCD that drives SWD/JTAG probes (ST-Link, J-Link, CMSIS-DAP) for flashing and debugging, with a GDB server and VS Code integration. See GDB for Embedded.
  • defmt — a deferred-formatting logger: the device sends a compact ID + raw args over RTT and the host does the string formatting, so logging costs a fraction of a printf and barely touches the firmware’s flash/timing budget.
  • cargo + crates.io — one build system and dependency manager, replacing the Make/CMake/PlatformIO sprawl. Cross-compilation is a target triple (thumbv7em-none-eabihf) plus a .cargo/config.toml.

Where this connects

  • Rust — ownership, borrowing, traits, and async that this page assumes as background.
  • Startup Code & C Runtime — cortex-m-rt is its Rust counterpart: reset vector, .data/.bss init before #[entry].
  • CMSIS — the same SVD files feed both CMSIS headers and svd2rust-generated PACs.
  • Interrupts & Ring Buffers — ISR-shared data, handled here by Mutex/critical sections instead of volatile.
  • RTOS, priority inversion, tickless idle — RTIC and Embassy occupy the scheduling niche a kernel fills in C.
  • MISRA C & Defensive Firmware — many MISRA rules (no dynamic alloc, no aliasing, checked conversions) are language-enforced in Rust.
  • RTT & Semihosting / GDB for Embedded — defmt and probe-rs are the Rust-native take on probe-based logging and debug.
  • ESP32 — Espressif ships esp-hal/esp-idf-hal, making Xtensa/RISC-V Rust a first-class, vendor-supported path.

Pitfalls

  1. Reaching for alloc out of habit. A heap on an MCU brings fragmentation and nondeterminism; prefer heapless fixed-capacity collections unless you have a clear reason.
  2. Fighting the borrow checker on shared state instead of using the right tool. ISR-shared data wants Mutex<RefCell<…>> + critical section, or a framework (RTIC/Embassy) — not static mut, which is unsafe and now lints hard.
  3. Assuming unsafe disappeared. DMA buffers, cache maintenance, and bare register access still need it; the win is keeping that surface small and reviewed, not zero.
  4. Ignoring monomorphization bloat. Instantiating large generics across many types can blow the flash budget; measure .text.
  5. Treating defmt logs like printf. Formatting happens host-side, so logging a value the host can’t decode (mismatched defmt versions) yields garbage; keep host and target tooling in lockstep.
  6. No stack-overflow guard. Rust doesn’t bounds-check the stack; without flip-link/MPU an overflow is the same silent corruption as in C.
  7. Mixed maturity across vendors. STM32, nRF, RP2040, and ESP32 HALs are strong; obscure parts may have thin or no crates — check the ecosystem before committing a board.
  8. Expecting C-identical codegen. Bounds checks and panic paths add a little code/size; usually negligible, occasionally relevant in the tightest ISR — profile rather than assume.
